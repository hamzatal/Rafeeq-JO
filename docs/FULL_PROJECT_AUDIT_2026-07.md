## 2) 🎨 مشكلة التصميم — التشخيص الدقيق (جوهر شكواك)

المشكلة ليست أن التصميم «سيّئ»، بل أن **المشروع يشحن 3–4 أنظمة تصميم في آنٍ واحد** بسبب إعادة كتابة الهوية 7 مرّات دون حذف القديم:

| الطبقة | الملف | الهوية الفعلية |
|--------|-------|----------------|
| **runtime الموبايل (المصدر الحقيقي)** | `packages/shared/src/theme/scheme.ts` | **v7 "Onyx"** = أسود حبري `#0F1216` + أزرق `#2F6BFF` |
| ثوابت static (سبلاش/أخطاء/خرائط) | `packages/shared/src/theme/colors.ts` | **v6 "Navy+Gold"** = كحلي `#243B7A` + ذهبي `#E7A63A` |
| لوحة الأدمن كاملة | `admin-dashboard/tailwind.config.ts` + `globals.css` | **v6 Navy+Gold** + حيلة `cyan-* → gold` |
| شاشات المصادقة الموبايل | `AuthShell.tsx` (الطالب+الكابتن) | كحلي مثبّت يدوياً `#141B2E` |
| التوثيق | `docs/DESIGN_SYSTEM.md` | يقول v6 Navy+Gold (متأخّر عن الكود) |

### الأثر الملموس على المستخدم
- **رحلة الطالب تعبر 3 هويّات:** سبلاش كحلي/ذهبي ← ترحيب أسود Onyx ← تسجيل/OTP **أبيض فاتح** ← دخول **كحلي** ← التطبيق أسود/أزرق.
- **التطبيقان لا يتطابقان:** خلفية ترحيب الطالب `#0A0D12` (Onyx) بينما الكابتن `#141B2E` (كحلي).
- **الأدمن كأنه منتَج آخر:** كحلي/ذهبي + كلمة «Rafeeq JO» لاتينية ذهبية، مقابل موبايل أسود/أزرق + «رفيق» عربية.
- **شاشة الخطأ (`ErrorBoundary`) والسبلاش تستخدمان الذهبي**، فتظهر لمسة ذهبية لا وجود لها في باقي التطبيق.
- تعليقات قديمة مضلِّلة منتشرة: «soft lime glow»، «navy+cyan»، «v3»، «v6 Navy+Gold» — بقايا إصدارات ملغاة.

### التكرار في المكوّنات
- ملفات **متطابقة حرفياً (byte-identical)** بين الطالب والكابتن: `LiveMap.tsx` (~450 سطر!)، `Feedback`, `Icon`, `Loader`, `MapBackdrop`, `Screen` → صيانة مضاعفة.
- مكوّنات **تباعدت بصمت**: `TabBar` (الطالب = حبّة ملوّنة + badges، الكابتن = مؤشّر علوي رفيع)، و`ui.tsx` (بطاقات الطالب بلا حدود/زوايا أكبر/ظل md، الكابتن بحدود/أضيق/ظل sm، والكابتن يفتقد `ScreenHeader`/`ServiceTile`).

### الخلاصة التصميمية
لا يلزم نظام تصميم ثامن. يلزم **الالتزام بنظام واحد وحذف البقية**. الأفضل هو تثبيت **v7 "Onyx"** (هو بالفعل مصدر الـ runtime لأكبر وأنظف سطح = تطبيق الموبايل الرئيسي)، لأن الإصلاح غالبيته **حذف** لا كتابة.

## 3) 🔒 الأمان والجودة (QA) — أهم النتائج

> التفاصيل الكاملة بمراجع الملفات في `backend/SECURITY_QA_AUDIT.md`.

### 🟥 حرجة — حواجز إطلاق (يجب سدّها قبل الأموال الحقيقية)
1. **قنوات البثّ اللحظي عامّة بلا تفويض** — `TripLocationUpdated`/`TripStatusChanged` تبثّ على `Channel('trip.{uuid}')` عام، ولا يوجد `channels.php`/`Broadcast::channel()`. أي شخص يعرف UUID الرحلة يتتبّع موقع GPS للطالب لحظياً — خطر خصوصية/سلامة خطير (خاصة للطالبات). **الحل:** `PrivateChannel` + تفويض القناة للكابتن + ركّاب الرحلة فقط.
2. **حجوزات المحفظة اليتيمة** — `TripService::start()` يحجز الأجرة، لكن `end()` لا يحرّر الحجز للراكب الذي حجز ولم يصعد → **رصيد الطالب مجمّد للأبد**. `cancel()` يحرّر، `end()` لا.
3. **تجاوز المصادقة الثنائية (MFA)** — الدخول بـ OTP بلا كلمة مرور (`verifyOtp` مسار Login) يُصدر توكن بلا فحص MFA، فيتخطّى موظّف الإدارة العامل الثاني.
4. **الكوبون قد يخلق رصيداً وهمياً** — في `RideBillingService::chargeForBoarding`، الكابتن يُمنح `fare - commission` دائماً بينما الطالب يدفع `fare - discount`؛ عند `discount > commission` يزيد الرصيد بلا إيداع (لا توجد محفظة خزينة). **الحل:** حدّ الخصم بنسبة العمولة أو خصمه من محفظة منصّة.

### 🟧 عالية/متوسطة
5. **سباق استبدال الكوبون** — `redeem()` يعيد فحص الحدّ الكلي فقط تحت القفل؛ `per_user_limit`/`first_order_only` بلا قفل → تجاوز متزامن.
6. **SOS بلا throttle ولا تحقّق ملكية `trip_id`** — و`throttle:sensitive` غير مطبّق على إنشاء الدفع/الإيصال رغم التعليق.
7. **فهارس ناقصة** على أعمدة FK/الاستعلامات الساخنة: `trip_passengers.student_id`, `paid_at`, `payments.bank_reference` → seq-scan في التقارير المالية مع نمو البيانات (Postgres لا يفهرس FK تلقائياً).
8. **حدود المعاملات** — `end()`/`cancel()` ينفّذان تحديثات متعددة خارج `DB::transaction` → احتمال حالة جزئية عند فشل.
9. **كود OTP الشامل/التصحيح** محميّ فقط بـ `APP_ENV !== 'production'` → خطأ إعداد واحد يكشف الـ OTP. أضِف حارساً صلباً.

### ✅ نقاط قوة مؤكّدة (لا إجراء)
- **لا حقن SQL** — كل الأجزاء الخام مربوطة بمعاملات.
- **لا mass-assignment** — `$fillable` صريح، `$request->validated()`، لا تصعيد صلاحيات في تحديث الملف.
- **لا IDOR** في مسارات الأموال المفحوصة — `assertOwner`/`ownedTrip` + مسارات الأدمن محميّة بصلاحيات.
- **رفع الملفات آمن** — قرص `secure` خاص، أسماء مُعمّاة، تحقّق mime/size، تنزيل للأدمن فقط، بصمة صورة للتكرار.
- **تزامن المحفظة/الدفع** — `lockForUpdate` + إعادة فحص داخل المعاملة، واعتماد الدفع idempotent مدعوم بفهرس فريد جزئي.
- **رؤوس أمان/CORS/throttle** معقولة، وبروت-فورس OTP غير عملي (6/دقيقة + كود مُعمّى).
- **TOTP** تطبيق RFC-6238 سليم.

## 4) ⚡ الأداء والقابلية للتوسّع (تحمّل ضغط المستخدمين)

- **Queues (Redis):** العمل الثقيل مؤجّل (تحقّق GPT-Vision، الإشعارات/الـ push)، والاستجابة تعود بسرعة — جيد.
- **مسارات تحبس الـ thread:** `SosService` يرسل SMS متزامناً + يحمّل كل الموظفين؛ `MatchingService::formTrips()` يحمّل كل الطلبات المعلّقة بالذاكرة (على جدول كل 5 دقائق، ليس على مسار الطلب). عند التوسّع: قوائم انتظار مخصّصة + تقطيع حسب الزون.
- **Pagination** مطبّق. **Caching** للصلاحيات لكل طلب.
- **الفهارس الناقصة (البند 7 أعلاه)** أهم عنق زجاجة مستقبلي للتقارير.
- **الجغرافيا:** Haversine في PHP + مطابقة بالمساواة (`zone_id`) — كافٍ الآن؛ إن احتجت بحث «أقرب كابتن» بمقياس كبير، فعّل PostGIS + فهارس GiST بدل مسح lat/lng.
